# Security Audit: Project Hydra

This document outlines the zero-trust security architecture embedded within the Hydra Protocol.

## 1. System Key Isolation (Orchestrator Trust)
The .NET backend acts as an Orchestrator. It authenticates users via their Web3 Ethereum Wallets (SIWE) and validates ECDSA signatures. When a user broadcasts a message, the `.NET` API signs the relay payload using a highly secured `SystemKey` (HMAC-SHA256).

**Vulnerability Prevented:** The C++ Ledger Node exposes a raw TCP socket (`8333`). Without the System Key signature, any malicious actor could bypass the Web3 verification and pipe junk data directly into the node. The C++ Node validates the `system_sig` prefix, ensuring it only accepts traffic from the verified .NET Orchestrator.

## 2. The Nonce Defense (Double-Broadcast Protection)
Every Broadcast Payload generated by the frontend includes a `nonce` (typically the timestamp) blended into the `content_hash`. 
- `contentHash = keccak256(content + timestamp)`
- The user signs this exact `contentHash`.

**Vulnerability Prevented:** A "Double-Broadcast" or "Replay Attack" occurs if an attacker intercepts a signed payload and re-submits it to the network to spam the ledger. Because the `content_hash` incorporates the timestamp/nonce, the C++ Node (and the MongoDB Aggregator) enforces unique hash constraints. If an identical hash is submitted twice, it is rejected at the index level, neutralizing replay attacks instantly.

## 3. Read-Only Aggregator Locks
The .NET `LedgerTailingWorker` uses `FileShare.ReadWrite` to access `ledger.dat`.
**Vulnerability Prevented:** Standard file operations lock the binary stream. By aggressively enforcing a read-only tailing mechanism, the .NET backend can never accidentally lock the C++ Node from appending new blocks, preserving the 24/7 uptime of the consensus engine.
